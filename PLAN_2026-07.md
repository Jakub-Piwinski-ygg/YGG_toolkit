## Etap 0 — oryginalne notatki

### B1. Eksport wideo walczy z timeline/scenario o viewport 🔴
- **Objaw**: przy odpalonym timeline/scenario eksport miga — dwa systemy renderują na zmianę.
- **Przyczyna (zlokalizowana)**: `PixiViewport.exportVideo()` (PixiViewport.jsx:446) zatrzymuje

  (PixiViewport.jsx:337-363) robi konkurencyjne `applyFlowAtTime` + `app.render()`
  w innym czasie niż eksporter.
- **Fix**: (a) otwarcie `WebMExportDialog` zatrzymuje playback (flow, scenario, spine tick);
  (b) jedna flaga `exportActive`/`playbackSuspended` w SceneStudioInner honorowana przez
  wszystkie trzy pętle; (c) `finally` przywraca stan.
- Rozmiar: **S/M**.

### B2. Number inputy clampują w trakcie pisania 🔴 (najbardziej irytujące)
- **Przyczyna**: `DragNumberField.jsx:104-109` — trzyma raw text, ale i tak woła
  `onChange(clamp(v))` na każdy keystroke (wpisanie "1" przed "20" → clamp do min).
  Dodatkowo dziesiątki gołych `<input type="number">` z inline `Math.max/min` w onChange:
  WinSequenceWizard (778-847), SpinnerWizard, TimelinePanel, ScenarioInspectorSections,
  UnityExportDialog, CheatTool (`FieldRow.jsx` + sections), Scaler/Crop/Converter/Atlas/
  GaussianBlur/Blur/AssetLibrary/Paylines/FontPreview, UnityExportConfigPanel.
- **Fix**:
  1. `DragNumberField`: podczas edycji z klawiatury commit+clamp **dopiero na Enter/blur**
     (Esc-revert już jest); drag-scrub może clampować live (nie irytuje).
  2. Nowy współdzielony `NumberField` (ten sam kontrakt commit-on-Enter/blur) i migracja
     wszystkich gołych inputów na niego.
  3. Pola wymagające live-update: aktualizować tylko gdy wpisany tekst parsuje się do
     wartości w zakresie; nigdy nie nadpisywać tekstu użytkownika w trakcie pisania.
  4. ✅ Zasada dodana do `CLAUDE.md` (What NOT to do) + `brain/10-Architecture/Gotchas.md`
     — obowiązuje każde nowe pole.
- Rozmiar: **M** (mechaniczna migracja, dużo call sites).

### B3. Czasem nie da się move/scale/rotate na scenie 🔴
- **Kandydaci (z audytu, wszystkie realne)**:
  1. `getObjectMetrics` → null gdy `getLocalBounds()` 0×0 (Spine mid-load, brak
     `__baseBounds`) → obiekt nieklikany (`pixiApp.js:479`, `viewportController.js:621`).
  2. Okno structural-rebuild (PixiViewport.jsx:295-332) — handles destroyed/swapped.
  3. `selectedLayerId` zerowane gdy `wizardActive || studioMode === 'direct'`
     (PixiViewport.jsx:3386) → brak uchwytów resize/rotate.
  4. Playback nadpisuje drag co klatkę: `syncTransforms` resetuje x/y/scale/rotation
     podczas odtwarzania, a drag commituje dopiero na mouseUp.
- **Fix**: cache bounds fallback dla (1); podczas aktywnego draga viewport ma priorytet
  nad `syncTransforms` dla przeciąganego obiektu (4) — spina się z flagą z B1;
  diagnostyczny log do app-log żeby przyszłe przypadki dały się nazwać.
- Rozmiar: **M**.

## W2 — Analiza NUX flow tworzenia sceny (2026-07-02)

Przejście pełnego flow nowego użytkownika po zmianach z tego batcha. Format:
**tarcie → rekomendacja (rozmiar)**. To feeduje kolejny batch — nie jest jeszcze
zaimplementowane.

### Flow jaki jest teraz
1. Wejście `?tool=scenestudio` → **start screen** (`WorkspaceLockOverlay`) zasłania
   wszystko: pick local folder / remote repo / drop.
2. Po załadowaniu: lewy panel (hierarchy + 🧰 toolkit + workspace assets), środek
   (viewport), prawy (inspector), dół (timeline w animate).
3. Tworzenie: Toolkit → 🎬 Scene Setup / 🎰 Spinner / 🏆 Win Sequences; statyki
   klik/drag z workspace.
4. Tryby setup / animate / direct; eksport WebM / Unity.

### Tarcia i rekomendacje (priorytetowo)

**P1 — największy zwrot dla NUX**
- **Brak „pierwszego kroku".** Po załadowaniu workspace scena jest pusta i user nie
  wie, że ma zacząć od 🎬 Scene Setup. → *Gdy scena pusta, auto-podświetlić/otworzyć
